Repository Analysis · LLM-Wiki · JanusX

把 agentmemory 的记忆系统，转译成 JanusX 的 LLM-Wiki 引擎

agentmemory 不是一个简单的“把聊天记录存起来”的项目。它的核心价值在于：保留原始事实源，把 LLM 作为抽取、压缩、归纳和图谱构建器，再用 BM25、向量和知识图谱三路检索把高价值上下文注入下一次工作。

53 MCP 工具，覆盖记忆写入、召回、图谱、治理、导出等操作面
4 工作记忆、情节记忆、语义记忆、程序记忆四层维护模型
3 BM25、Vector、Graph 三路检索，经 RRF 融合

agentmemory 的核心抽象

Capture
Hooks / MCP / REST 从工具调用、用户 prompt、session 生命周期自动捕获事件。
Fact Source
Raw Observation 原始观测先落库，后续摘要、索引、图谱都可重建。
LLM Jobs
Compress / Summarize / Extract LLM 负责结构化，不直接成为唯一事实源。
Indexes
BM25 + Vector + Graph 关键词、语义、实体关系互补，避免单一检索失效。
Viewer
Live Stream + Graph + Replay 把记忆维护变成可观察、可审计、可修复的系统。
01 · Repository Reading

这个仓库解决的不是“存储”，而是记忆生命周期

01
自动捕获 PostToolUse、UserPromptSubmit、Stop、SessionStart 等 hook 把行为转成 observation。
02
去重与隐私过滤 5 分钟窗口 SHA-256 去重，写入前剥离 secrets、API keys 和 private 标签。
03
原始观测落库 先存 raw observation，再派生 synthetic 或 LLM compression。
04
LLM 压缩与总结 输出 facts、concepts、files、narrative、importance，并支持 chunk map-reduce。
05
索引与图谱 BM25、vector、graph nodes/edges 都是派生层，可重建、可迁移。
06
召回与注入 SessionStart 或显式搜索按 token budget 注入最相关上下文。

这套设计最值得借鉴的点是“不可逆事实”和“可重算智能层”的分离。LLM 输出有概率错误，所以 agentmemory 没有把压缩结果当成唯一真相，而是让原始观测、派生摘要、图谱、索引、审计记录各自独立存在。

02 · Strong Designs

优秀设计特点：它把记忆工程化，而不是把 prompt 写长

1. 多入口，单后端

MCP、REST、hooks、CLI 都进入同一套函数注册体系。外部 Agent 的差异被压在适配层，记忆核心保持统一。

src/mcp/server.ts src/triggers/api.ts src/hooks/*.ts

2. LLM 可选，系统仍可用

没有 LLM key 时仍走 synthetic compression 和 BM25；配置 provider 后才启用 LLM 压缩、总结、图谱提取。这比“强依赖云模型”更适合本地工具。

src/functions/observe.ts src/functions/compress-synthetic.ts

3. 记忆有版本与替代关系

mem::remember 对相似内容做 Jaccard 检测，旧记忆标记为非 latest，新记忆记录 parent/supersedes。这让知识库能自然演化，而不是不断追加重复条目。

src/functions/remember.ts src/functions/cascade.ts

4. 图谱读取为可视化做了专门优化

大图不再依赖全量 kv.list，而是维护 name index、edge key、degree 和 top-degree snapshot。viewer 常用路径读快照，避免 75K 节点级别的超时。

src/functions/graph.ts src/state/schema.ts

5. 检索不是单一路线

BM25 负责精确词，向量负责语义相似，图谱负责实体邻近；最后用 RRF 融合，并保留每条结果的来源分数和图谱上下文。

src/state/hybrid-search.ts src/functions/smart-search.ts

6. 可观测性是产品能力

viewer 展示 live observation stream、session explorer、memory browser、graph visualization、health dashboard。记忆系统不再是黑箱。

src/viewer/index.html src/viewer/server.ts
03 · LLM-Wiki Principle

你的理想机制可以成立，但要让 LLM 写“候选补丁”，不要让它直接改真相

LLM-Wiki 的关键不是让模型“记住一切”，而是让模型持续把事件提炼成可检索、可审计、可回滚的知识单元。

JanusX 已经接入底层 LLM，因此不需要再引入一个外部记忆服务作为强依赖。更稳的路线是：吸收 agentmemory 的数据模型和维护机制，在 JanusX 主进程里实现本地知识库服务。

事实源
保留 agent session、checkpoint、git diff、用户 prompt、LLM 响应、工具事件等原始记录。
LLM 输出
产出 CandidateWikiPatch、facts、entities、relations、confidence、source refs。
应用策略
高置信自动应用到 draft wiki；低置信进入 review queue；冲突项必须人工确认或二次验证。
审计
每次新增、修改、删除、合并都记录来源 observation、模型、prompt、时间和操作者。

05 · Reference, Not Migration

借鉴的是知识库设计原则，不是照搬它的运行时

agentmemory 的优秀设计 JanusX 应该吸收什么 JanusX 不应该照搬什么 高效实现方式
Hooks 自动捕获工作过程 把终端、Agent、checkpoint、git analyzer、JanusChat 的关键事件统一成 knowledge observation。 不要另起一套 hook runtime；JanusX 已有 IPC、AgentStreamManager、TerminalManager、Analyzer。 新增 KnowledgeCaptureService，订阅现有事件并写入本地 JSONL。
Raw observation 与派生层分离 原始事件不可丢；facts、wiki、graph、index 都可重算。 不要让 LLM 直接覆盖 Markdown Wiki，把 Wiki 当唯一真相。 采用 Observation -> Candidate -> Reviewed Fact/WikiPatch 三段式。
LLM 压缩、总结、图谱抽取 复用 JanusX 现有 LlmService 和 AI SDK generateObject 路径。 不要引入新的 provider 配置、环境变量体系或模型调用层。 新增 KnowledgeExtractService，system prompt 与 schema 独立维护。
BM25 + Vector + Graph 融合检索 MVP 先做 BM25 + 标签过滤；确认使用频率后再加 embedding 和 graph ranking。 不要一开始做完整 RRF、reranker、graph traversal、query expansion。 用轻量倒排索引和 tag/workspace/file 过滤，接口保持可扩展。
Viewer 让记忆系统可观察 做进 JanusX 的面板：Knowledge Workbench、Graph、Tag Manager、Audit。 不要另起 Web viewer、端口、鉴权、CSP、代理层。 复用 Electron Renderer、Monaco、ReactMarkdown、@xyflow/react。
MCP 暴露记忆能力 给 Agent 工作流暴露少量高价值工具：capture、recall、context、patch、audit。 不要复制 53 个工具，工具面过大反而增加 Agent 选择成本。 先做 6-8 个核心 MCP tools + 2-3 个 resources/prompts。

评估结论：JanusX 的最佳方案是“内嵌式 Knowledge Engine”。它借鉴 agentmemory 的生命周期、派生层、审计、检索和可视化原则，但运行在 JanusX 已有主进程、IPC、LLM、Agent 和蓝图体系内。这样能避免双运行时、双配置、双 UI、双记忆模型。

A
现有事件进入 Terminal input、Agent event、checkpoint、git commit diff、Janus Analyzer result、用户手动保存。
B
规范化为 Observation 补齐 workspace、repo、file、agent、blueprint node、source tags。
C
先写事实源 追加到 .janusX/knowledge/observations.jsonl，不等待 LLM。
D
异步生成候选 LLM 只产出 facts、tags、wiki patch、graph edges 的候选结果。
E
规则层应用 高置信低风险自动入库；冲突、低置信、跨 workspace 影响进入 Inbox。
F
工作流消费 JanusChat、xlocal、Analyzer、MCP tools 从统一 context builder 取上下文。

最小核心

KnowledgeStore、KnowledgeCaptureService、KnowledgeExtractService、KnowledgeSearchService 四个服务足够启动闭环。

存储取舍

MVP 沿用 .janusX 目录约定：observations.jsonl、facts.jsonl、wiki/*.md、index/*.json。先不引入 SQLite。

UI 取舍

只做一个 Knowledge Workbench，不新增独立 viewer。先把 Inbox、Wiki、Tags、Graph、Audit 放成 tabs，避免多入口分散。

06 · JanusX Fusion Slice

把优秀设计压缩成 JanusX 里一个小而完整的知识闭环

推荐第一版只证明一件事：JanusX 能从真实工作流中捕获事实，用现有 LLM 生成候选知识，并把可追溯、可审核、可召回的上下文还给 Agent。
JanusX 融合点 应该借鉴 agentmemory 的设计 第一版实现边界 不做什么
KnowledgeStore 原始事实源与派生层分离，所有智能结果可重算。 只管理 observations.jsonl、facts.jsonl、wiki/*.md、indexes/*.json。 不引入数据库、不做多租户、不做服务端同步。
KnowledgeCaptureService 多入口归一化成 observation。 订阅 AgentStreamManager、checkpoint、Analyzer、JanusChat 的关键事件。 不新增 hook runtime，不改造现有 Agent 事件总线。
KnowledgeExtractService LLM 做抽取、总结、关系候选，不直接改真相。 复用 LlmService 和结构化输出，产出 CandidateFact、CandidateWikiPatch、CandidateGraphEdge。 不复制 provider 配置，不接入第二套模型 SDK。
KnowledgeSearchService 检索层可以逐步演进。 先做 BM25 / 倒排索引 + workspace/tag/file 过滤 + source refs 返回。 不在 MVP 做 embedding、reranker、RRF、query expansion。
KnowledgeWorkbench Viewer 的价值是可观察、可修正、可审计。 一个面板内放 Inbox、Wiki、Tags、Graph、Audit tabs。 不做独立 Web viewer，不开端口，不做第二套 UI shell。
KnowledgeMcpAdapter MCP 让外部 Agent 读写记忆。 只暴露 capture、recall、context、patch、tag、graph、audit 等少量工具。 不复制大而全的工具面，不让 Agent 在 50+ 工具里选择。

第一版目录建议

src/main/knowledge/* 放 store、capture、extract、search；src/main/ipc/knowledge-handlers.ts 放 IPC；renderer 只新增一个 Workbench 入口。

工作流接入矩阵

xunit 只取当前文件/任务知识；xlocal 取当前模块和近期决策；xwhole 取跨模块图谱、历史架构约束和冲突队列。

验收标准

一次真实任务结束后，系统应能生成至少 1 条 fact 候选、1 个 wiki patch、可通过标签召回，并能从 UI 反查原始 observation。

这里的关键取舍是“嵌入 JanusX 主工作流”，不是“再造一个知识库产品”。只要 capture、candidate patch、review、recall 四件事闭环成立，后续图谱、向量和 MCP 增强都有清晰落点；如果一开始追求完整平台，反而会偏离 JanusX 的 Agent 工作流目标。

07 · Data Model

最小可行数据模型：先能维护，再追求智能

对象 核心字段 用途 是否由 LLM 直接写入
Observation id, source, sessionId, projectPath, eventType, payload, createdAt, privacyFlags 事实源，来自 agent stream、checkpoint、git diff、chat。 否
MemoryFact id, content, concepts, files, tags, confidence, sourceObservationIds, version, supersedes 可检索事实，支撑 prompt 注入和 wiki 页面生成。 生成候选，规则层落库
KnowledgeTag id, name, kind, aliases, color, createdBy, mergedInto, usageCount 标签字典，区分系统标签和用户标签，支持别名、合并和统计。 可建议，不自动重命名
WorkspaceScope workspaceId, rootPath, repoRemote, defaultTags, visibility, trustLevel 记录知识来源工作区，控制跨工作区召回、权限和默认标签。 否
WikiPage slug, title, markdown, sourceFactIds, updatedAt, status LLM-Wiki 的可读层，供人和 Agent 消费。 生成 patch，不直接覆盖
GraphNode / GraphEdge type, name, properties, sourceObservationIds, weight, confidence 可视化、跨页面跳转、图谱召回。 生成候选，合并器校验
AuditEvent operation, targetId, actor, model, promptHash, before, after, createdAt 回滚、追责、解释“这个知识从哪来”。 否
08 · Tags & Workspaces

知识单元必须跨工作区流动，但不能丢失标签语义

#type/fact #type/decision #type/workflow #type/preference #workspace/janusx #workspace/client-a #repo/packages-llm-core #file/src/main/llm #agent/coderX #mode/xlocal #task/checkpoint #status/draft #status/verified #lifecycle/superseded #confidence/high #visibility/private #visibility/team

标签不是装饰，而是召回和治理维度

JanusX 的知识单元应该允许来自任意工作区：项目 A 的 LLM Provider 经验、项目 B 的 Agent 流程、某个客户仓库的架构约束，都能沉淀到统一知识库。但每条知识必须带上工作区、来源、类型、置信度、生命周期和可见性标签。

推荐采用“系统标签 + 用户标签”双轨：系统标签由 JanusX 自动生成并参与权限、检索、过期和冲突判断；用户标签用于个人知识组织，保留 Obsidian 式自由度。

系统标签
#workspace/*、#repo/*、#type/*、#source/*、#confidence/*、#visibility/*。
用户标签
例如 #架构、#提示词、#复盘、#坑位。允许别名、合并、重命名，但必须保留审计。
召回策略
默认优先当前 workspace；显式查询可跨 workspace；高风险知识必须同时匹配 workspace/repo/file 标签。

一个合理的知识单元不是一段裸文本，而是 content + tags + source refs + confidence + lifecycle。标签让它能跨工作区复用，source refs 让它能被追溯，lifecycle 让它能被替代和清理。

09 · Visual Management

可视化管理不是附属 UI，而是知识库可靠性的控制台

Inbox

展示未处理 observations、LLM 候选 facts、候选 wiki patch。按置信度、来源、项目、时间排序。

Wiki Editor

左侧页面树，中间 Markdown 编辑，右侧 provenance。所有 LLM 修改以 diff 形式呈现。

Knowledge Graph

用 @xyflow/react 呈现概念、文件、模块、决策、Agent、任务之间的关系。

Conflict Queue

当新事实与旧事实冲突、版本替代、文件 truth 不一致时进入人工确认队列。

Search Lab

展示 BM25、vector、graph 各自命中和融合结果，便于调权重、查召回失败。

Maintenance

提供重建索引、重跑抽取、合并重复、归档 stale 页面、导出 Markdown 的操作入口。

10 · Obsidian Graph

做像 Obsidian 的知识图谱，但节点语义要服务 Agent 工作流

LLM Provider#type/concept
JanusX#workspace/core
Checkpoint#type/workflow
LlmService#file/main
Vertex AI#provider
coderX#agent

图谱节点类型

建议至少区分 WikiPage、Fact、Concept、File、Workspace、Agent、Task、Decision、Prompt。不要只做“页面互链”，否则图谱只能看，不能驱动工作流。

图谱边类型

边应表达具体关系：mentions、derived_from、supersedes、depends_on、implemented_in、owned_by、conflicts_with、used_by_agent。

交互能力

支持标签过滤、workspace 过滤、时间滑块、置信度阈值、孤岛节点检测、冲突边高亮、从节点反查 source observation 和相关 prompt。

Obsidian 图谱的体验重点是“发现关系”。JanusX 图谱还要承担“给 Agent 提供上下文”的职责：选中图谱子图后，应能一键生成 prompt context、handoff context、review context 或 MCP resource。

11 · MCP Surface

MCP 扩展要把知识库变成 JanusX 工作流的一等能力

knowledge_capture

Agent 在完成任务、读文件、遇到错误、做架构决策时主动写入 observation 或 fact candidate。

knowledge_recall

按 query、workspace、tags、files、agent role 召回相关知识，返回带引用的 compact context。

knowledge_context

为 xwhole/xlocal/xunit 自动生成上下文包：相关 wiki、历史决策、注意事项、最近冲突。

wiki_patch_propose

让 LLM 基于当前任务产出 Wiki patch，但进入 review queue，不直接覆盖页面。

wiki_patch_apply

应用已确认 patch，写入审计事件，更新 page version、tags、graph edges 和索引。

tag_query

按系统标签和用户标签查询知识单元，支持别名、合并历史和标签统计。

graph_query

查询 Obsidian 式子图，支持 entity、tag、workspace、depth、edge type、confidence 过滤。

knowledge_audit

查询某条知识从哪里来、被谁改过、由哪个模型生成、与哪些文件和任务相关。

这些 MCP 工具应同时暴露给 JanusX 内置 Agent、外部 Codex/Claude/Cursor、以及用户脚本。真正的集成不是“Agent 能搜索知识库”，而是每个工作流阶段都能自动读、写、更新、审计知识库。

12 · Roadmap

落地路线：先本地可控，再逐步增强检索和图谱

Phase 1 · 本地事实源、标签和 Wiki 草稿

建立 .janusX/knowledge 存储；从 AgentStreamManager、checkpoint、git analyzer 写 observations；知识单元强制带 workspace/source/type 标签；用 LlmService 生成 facts 和 Wiki patch；只做 BM25 搜索。

Phase 2 · 可视化工作台

实现 KnowledgeWorkbench：Inbox、Wiki Editor、Tag Manager、Provenance、Audit Log。让用户能看到并修正 LLM 的每一次知识更新。

Phase 3 · Obsidian 风格图谱与关系召回

抽取实体与关系，接入 @xyflow/react；支持标签过滤、workspace 过滤和 source 反查；学习 agentmemory 的 side-index + snapshot 设计，不让图谱页面依赖全量扫描。

Phase 4 · 向量检索与混合排序

复用 llm-core 的 embedding 能力或新增 embedding model；用 RRF 融合 BM25、vector、graph，展示每路得分。

Phase 5 · MCP 工作流闭环

暴露 knowledge MCP tools/resources/prompts，让 xwhole/xlocal/xunit 自动读写知识；增加 TTL、访问强化、冲突检测、版本替代、定期摘要、Markdown 导出和项目间迁移。

13 · Cautions

直接集成 agentmemory 的风险，以及更适合 JanusX 的取舍

选择 收益 风险 建议
直接运行 agentmemory 功能完整，MCP/REST/viewer 现成。 引入 iii-engine、端口、进程、外部数据模型；与 JanusX LLM/agent/checkpoint 栈重复。 适合作为外部可选适配，不建议作为核心依赖。
移植核心机制 与 JanusX LlmService、Electron IPC、已有 UI 栈自然融合。 需要自己实现存储、索引、图谱维护和审计。 推荐。先做最小闭环，再补混合检索。
只做 Markdown Wiki 实现最快，人类可读。 缺少 provenance、冲突检测和检索质量；会退化成“LLM 乱改文档”。 可以作为消费层，但不能作为唯一存储层。
一开始做完整知识平台 规划看起来完整。 标签、图谱、向量、MCP、审计、自动维护同时开工，会拖慢 JanusX 主工作流并制造大量空接口。 不要做。先交付 observation + tags + candidate patch + BM25 + Workbench Inbox 的闭环。
