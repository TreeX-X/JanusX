# 飞书双向控制使用指南

## 1. 能力范围

JanusX 可以通过飞书自建应用的长连接接收消息和卡片动作，用于控制已明确绑定的本地 CLI Agent 终端（Claude Code、Codex、OpenCode）。JanusX 必须保持运行，目标终端也必须仍然存活。

已交付：状态查询、显式绑定/解绑、单行 follow-up、停止、批准、拒绝、签名卡片按钮、结果回执、去重和本地审计。

未交付：Janus Chat follow-up（P1b）、任意 shell/文件操作、随机终端回退、HTTP 公网回调、Companion Web、桌面或键鼠远控。Webhook 模式只发送通知，不能接收入站控制。

## 2. 飞书开放平台配置

1. 在飞书开放平台创建“企业自建应用”，记录 App ID 和 App Secret。
2. 在“应用能力 > 机器人”中启用机器人。
3. 在“权限管理”申请并由管理员批准以下应用身份权限：
   - `im:message:send_as_bot`：以应用机器人身份发送通知和回执。
   - `im:message.p2p_msg:readonly`：接收用户发给机器人的单聊消息。
   - `im:message.group_at_msg:readonly`：接收群聊中 @机器人的消息。
4. 在“事件与回调 > 事件配置”选择“使用长连接接收事件”，不要配置公网 Request URL。
5. 添加事件 `im.message.receive_v1`（接收消息）。
6. 添加回调 `card.action.trigger`（交互卡片动作）。
7. 创建并发布应用版本，把应用安装到目标企业；将机器人加入目标群聊或与机器人建立单聊。

事件名和发送权限与 JanusX 使用的官方 `@larksuiteoapi/node-sdk` 一致。修改权限、事件或回调后需要重新发布应用版本。

## 3. 获取 open_id 和 chat_id

- 操作者白名单必须填写飞书 `open_id`，格式通常为 `ou_...`。可在飞书 API 调试台查询用户，或从 `im.message.receive_v1` 的 `sender.sender_id.open_id` 获取。
- 群聊投递目标使用 `chat_id`，格式通常为 `oc_...`。可通过飞书 API 调试台的群列表接口查询，或从消息事件的 `message.chat_id` 获取。
- 单聊出站目标也可使用 `open_id`。但 JanusX 只有在 Receive ID Type 为 `chat_id`、目标终端明确且白名单只有一个操作者时才生成控制按钮；否则仍发送不带按钮的通知卡片。

API 调试台仅用于发现 ID；JanusX 不需要联系人或群列表权限来处理已经配置好的 ID。

## 4. JanusX 设置

打开“设置 > 通知 > 飞书”：

1. 开启“远程提醒”和“飞书提醒”。
2. 发送模式选择 `App`。
3. 填写 App ID、App Secret、Receive ID Type 和 Receive ID。
4. 开启“飞书双向控制”。
5. 每行填写一个允许操作的 `ou_...`。空值、无效值和重复值会被移除。
6. 按需要调整：
   - 绑定有效期：默认 480 分钟，范围 5 分钟到 7 天。
   - 卡片动作有效期：默认 10 分钟，范围 1 到 60 分钟。
   - 审计保留期：默认 30 天，范围 1 到 365 天。
   - 最大 follow-up：默认 4000 字符，范围 1 到 4000 字符。
   - 群聊 prompt 前缀：默认 `/p`，必须为非保留的 `/name`。
7. 保存。App Secret 输入框留空表示保留已保存值；main 进程不会把已保存 secret 返回给界面。
8. 点击状态旁的刷新按钮。正常启动依次可见 `connecting`、`connected`；错误只显示经过脱敏和截断的摘要。
9. “测试飞书通知”只验证出站发送；入站是否可用以连接状态和实际命令回执为准。

启用控制时，JanusX 会拒绝缺少 App ID/Secret、Receive ID 或有效 open_id 的设置。切换为 Webhook 或清空白名单会停止入站控制，不影响已配置的出站通知。

## 5. 使用命令

先在 JanusX 打开一个 CLI Agent 终端，取得界面中的 terminal ID。飞书命令必须完全匹配：

| 命令 | 作用 |
|---|---|
| `/status` | 查看当前飞书会话/话题是否绑定终端。 |
| `/terminals` | 列出已登记工作区和当前存活的 CLI Agent 终端，并返回 Bind / New Terminal 卡片控件。 |
| `/bind <terminal-id>` | 绑定一个明确存在的 CLI Agent 终端。不会选择“最近”或随机终端。 |
| `/unbind` | 删除当前会话/话题的绑定。 |
| `/stop` | 向已绑定终端发送中断。 |
| `/p <text>` | 把一行 follow-up 提交到已绑定终端；`/p` 可替换为设置中的群前缀。 |

单聊中，白名单用户发送的普通文本会作为 follow-up，但仍要求有效绑定。群聊中必须 @机器人或使用配置的前缀。多行、空文本、控制字符和超过长度限制的文本会被拒绝。

不知道 terminal ID 时，发送 `/terminals`。卡片会按已登记工作区分组，显示工作区名称、目录名、引擎和运行状态。直接点击目标终端的 **Bind**；如果工作区没有存活终端，可点击 **New Claude**、**New Codex** 或 **New OpenCode**，创建完成后 JanusX 会发送刷新卡片，再点击新终端的 **Bind**。新建只允许已登记且路径有效的工作区，不接受任意路径或 shell。

## 6. 卡片按钮与回执

App 通知卡可能包含 Bind、Stop、Approve、Reject：

- 按钮只绑定一个 chat、一个 terminal、一个动作、一个操作者和到期时间；不能转发给另一操作者使用，也不能重放。
- 白名单有多个操作者、Receive ID 不是 `chat_id`、缺少 terminal 上下文或入站未连接时，卡片不显示控制按钮。
- Approve/Reject 只在目标终端确实处于待批准状态时有效。

每次请求会收到简短回执：Accepted、Rejected、Expired、Unbound、Stale target、Duplicate 或 Failed。回执不包含 secret 或长终端输出。

## 7. 审计与吊销

审计文件位于 Electron userData 下的：

```text
janusx/companion/audit.jsonl
```

每项操作记录 intent/outcome、操作者、chat/thread、动作、目标 terminal 和结果。follow-up 只记录短预览、长度和 SHA-256，不记录 action token。记录按设置的保留天数清理。

立即吊销方式：

1. 关闭“飞书双向控制”，或切换为 Webhook 并保存。
2. 删除操作者 open_id；清空白名单会自动关闭入站控制。
3. 在飞书开放平台重置 App Secret、撤销权限、停用应用或移除机器人。
4. 在飞书发送 `/unbind` 可只撤销当前会话绑定。

## 8. 故障排查

- `disabled`：检查远程提醒、飞书 provider、App 模式和双向控制四个开关。
- `not configured`：补齐 App ID、App Secret 和 Receive ID；secret 留空只在以前成功保存过时表示“保持不变”。
- `error`：检查网络能否访问飞书开放平台、App ID/Secret、应用版本是否发布，以及长连接事件配置。
- 收不到消息：确认已添加 `im.message.receive_v1`，单聊/群聊读取权限已批准，群聊中已 @机器人或使用前缀。
- 按钮不出现：Receive ID Type 必须是 `chat_id`，通知必须带 terminal 上下文，且白名单必须恰好一个 open_id。
- 按钮提示过期/无效：重新等待一张新卡片；不要转发或重复点击旧按钮。
- `Unbound`：先执行 `/bind <terminal-id>`，并确认终端仍在 JanusX 中运行。
- `Stale target`：本地终端已退出，重新创建终端并绑定。
- 修改飞书权限或事件后无效：重新发布应用版本，再重启或重新保存 JanusX 设置触发重连。
