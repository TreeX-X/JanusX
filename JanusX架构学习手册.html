它到底是个什么东西

一句话：JanusX 是一个面向开发者的桌面工作台，它用 Electron 把多个 AI 编码 CLI、分屏终端、文件编辑、git、快照还原、蓝图分析、以及一个拟人化的「Janus 控制核心」装进一个无边框窗口里。

阅读建议

01 · Architecture

整体架构与技术栈

JanusX 是一个经典的 Electron 三层应用：主进程管「重活」（PTY、git、LLM、分析器），预加载层做安全桥接，渲染层是 React + Zustand 的 UI。下面这层关系是理解一切的根基。

monorepo 是怎么连起来的

用最轻量的 npm workspaces（不是 pnpm/turbo），根 package.json 声明 "workspaces": ["packages/*"]。根 workspace 本身就是 Electron 应用入口；唯一的子包 @janusx/llm-core 是可独立发布的 LLM 抽象层。

为什么不上 pnpm / turbo？

项目规模目前只有一个子包，npm workspaces 零配置够用；引入 turbo 会带来缓存配置与额外心智负担，不符合「一个子包」的现实。脚本里 npm run build:llm-core → npm run build --workspace=@janusx/llm-core，dev/build 先编译 llm-core 再跑 electron-vite。

IPC 是怎么按域名组织的

主进程在 src/main/index.ts 按领域注册 handler，每个领域一个文件，命名即语义：

IPC 文件对接服务典型频道
ipc/terminal-handlers.tsterminal/manager.ts (node-pty)terminal:create/input/resize/exit
ipc/agent-handlers.tsagent/stream-manager.tsagent:start/event/cancel
ipc/checkpoint-handlers.tscheckpoint/checkpoint-manager.tscheckpoint:create/list/restore
ipc/janus-handlers.tsjanus/analyzer.ts + blueprint-storejanus:analyzer:analyze
ipc/llm-handlers.tsllm/LlmService.ts + ConfigStorellm:config/chat
ipc/git-handlers.tsgit/service.tsgit:commit/log/status
为什么 sandbox:false

node-pty 这类原生模块在 sandbox 下无法直接 spawn 子进程；同时预加载层用 contextBridge + 显式频道白名单守住边界——安全靠「白名单 + 主进程持有能力」，而非靠 sandbox 隔离。这是 Electron 嵌入真终端时的常见取舍。

02 · LLM Core

LLM Core · 提供商抽象层

@janusx/llm-core 是一个可独立发布的包，职责非常窄：把「创建一个 LanguageModel」这件事，从具体厂商 API 里抽象出来。它本身不跑推理——推理交给 Vercel AI SDK 的 generateText/streamText。

三层架构

应用层 (JanusX main/llm/LlmService.ts)
        │  createLanguageModel(settings, modelId)
        ▼
门面层 (src/index.ts)  ── createLanguageModel / createEmbeddingModel / validateProviderConfig
        │  ProviderFactory.getInstance()  (单例)
        ▼
核心层 (core/)  ── ExtensionRegistry(注册表单例) + ProviderFactory(工厂单例) + types.ts(契约)
        │  resolveAdapter(authType) ── AUTH_TYPE_TO_ADAPTER 映射
        ▼
适配器层 (adapters/)  ── OpenAICompatibleAdapter / VertexAIAdapter
        │  动态 import('@ai-sdk/openai' | '@ai-sdk/google-vertex')
        ▼
Vercel AI SDK → 真实 provider HTTP

两个注册表，各司其职

运行时注册表 · ExtensionRegistry

单例，Map<string, ProviderExtension>。只存「适配器实例」。注意：包内不自动注册，消费方要自己 factory.register(new OpenAICompatibleAdapter())——注册表启动时是空的。

元数据目录 · ConfigLoader + providers.json

静态清单（version: 2026.06.16），给 UI 渲染表单用——字段类型、条件可见性、默认端点。不参与运行时实例化，可热重载。

工厂如何派发

packages/llm-core/src/core/ProviderFactory.ts:13-17const AUTH_TYPE_TO_ADAPTER: Record<string, string> = {
  [AuthType.API_KEY]:   'openai-compatible',
  [AuthType.VERTEX_AI]: 'vertex-ai',
  [AuthType.NONE]:      'openai-compatible',
}

createLanguageModel 走：缓存键 ${id}:${modelId} → 校验 → 解析适配器 → 调 adapter.createLanguageModel → 缓存返回。返回的对象就是 AI SDK 的 LanguageModelV1，调用方拿去喂 streamText。

OpenAICompatibleAdapter · 动态 import + baseURL 嗅探

packages/llm-core/src/adapters/openai-compatible/OpenAICompatibleAdapter.ts:62-91const { createOpenAI } = await import('@ai-sdk/openai')  // 动态导入，按需加载
const openai = createOpenAI({ baseURL: settings.baseURL, apiKey: settings.apiKey!, organization: settings.organization })
const model = openai(modelId) as any
return withAiSdkV1StreamCompatibility(model)   // 套一层流归一化

一个类服务所有 OpenAI 兼容厂商——靠 baseURL 区分。getDefaultModel 甚至直接嗅探 baseURL 子串：deepseek→deepseek-chat、moonshot→moonshot-v1-8k、zhipu→glm-4，否则 gpt-4o。

VertexAIAdapter · 四路鉴权 + 代理穿透

Vertex 的复杂度在 GCP 鉴权与代理。createLanguageModel 按 auth 模式四分支：useADC（应用默认凭证）/ clientEmail+privateKey / serviceAccountJSON（字符串解析）/ serviceAccountPath（keyFilename）。还自带 normalizePrivateKey() 处理用户粘贴 JSON 时 \n 转义、引号包裹等常见坑。代理通过 google-auth-library 的 transporterOptions.agent 注入。

stream-compat · 一个容易被忽略的关键设计

包钉了 specificationVersion: 'v1'，但厂商 SDK（@ai-sdk/openai@3、@ai-sdk/google-vertex@4）有时吐 v2 的 chunk 形状。withAiSdkV1StreamCompatibility 只拦截 doStream，用 TransformStream 逐 chunk 归一化：

  • text-delta：同时兼容 textDelta 和老 delta 字段
  • tool-call：统一成 {toolCallType:'function', toolCallId, toolName, args ?? input}
  • finish：promptTokens ?? inputTokens、completionTokens ?? outputTokens 双向兜底
  • v2 独有的 reasoning-* / tool-input-* 等 chunk 类型直接丢弃，不污染 v1 消费方
为什么要套这一层？

AI SDK 版本演进快，厂商 provider 版本与 ai 主包版本不完全同步。与其每次升级追着改消费代码，不如在适配器出口做一次「形状护栏」，让上层永远看到稳定的 v1 chunk。带幂等守卫（__janusxAiSdkV1StreamCompat 标记）防重复包裹。

ProxyManager · 进程级全局代理

utils/proxy.ts 的 ProxyManager 是个相当「侵入式」的单例：调用 configure() 后会 monkeypatch http/https.get/request 并设置 undici 全局 dispatcher 为 EnvHttpProxyAgent——一旦配置，进程内所有出站 HTTP 都走代理。Windows 上还能查注册表自动探测系统代理。

为什么这么激进？

国内访问 OpenAI / Vertex 普遍需要代理，而 AI SDK 内部的 fetch 不暴露每请求代理钩子。在进程入口一次性 hook，比要求用户在每个 SDK 调用处传 agent 务实得多——代价是全局副作用，所以放在最小依赖的 llm-core 里集中管理。

端到端请求流

App: createLanguageModel(settings, 'gpt-4o')
  └─► ProviderFactory.createLanguageModel
        ├─ cacheKey = "openai-compatible:gpt-4o"  命中则直接返回
        ├─ validateSettings (URL/Key 形状校验，不烧 quota)
        ├─ resolveAdapter(authType) → registry.get('openai-compatible')
        └─ adapter.createLanguageModel
              ├─ import('@ai-sdk/openai')
              ├─ createOpenAI(...)(modelId)
              └─ withAiSdkV1StreamCompatibility(model)   ← 流归一化
  ◄── 返回 LanguageModelV1

App: streamText({ model, messages })
  └─► AI SDK 调 model.doStream
        └─► 我们的 wrapper 拦截 doStream → TransformStream 归一化 → 上层消费 textStream
关键边界

llm-core 的职责止于产出 LanguageModelV1。真正发推理请求（generateText/streamText）是应用层用 ai 包发起的——包只拦截 doStream 做形状归一化。这条边界让包保持「薄而稳」。

03 · Terminal

Terminal · 分屏与 PTY

终端是 JanusX 的「执行现场」。它实现了一套 JetBrains 式的递归分屏树 + 标签页，背后是 node-pty 真终端，前后端用 Electron IPC 连。近期几个 commit（split UI、split drag、split preview）都在打磨这块。

数据结构 · 递归 leaf/split 树

src/renderer/src/lib/workspace-pane.ts:1-28export type WorkspacePaneLeaf = {
  type: 'leaf'; id: string
  tabs: PaneContent[]        // 一个 pane 里可挂多个终端 tab

}
export type WorkspacePaneSplit = {

核心操作 · 一套纯函数代数

workspace-pane.ts 是一棵树的纯函数代数——所有变更返回新节点，不就地修改。这让 Zustand 的 set({ paneTree }) 触发廉价重渲染：

函数行作用
splitPaneTree279-323把目标 leaf 包进新 split + 空兄弟 leaf
collapsePaneTree325-363整树拍平成单 leaf，去重终端
unsplitPaneTree434-447移除一个 leaf，tab 合并到存活兄弟
resizeSplitPane449-472更新 ratio，钳制 [0.15, 0.85]
pruneEmptyPanes82-97编辑后清理空叶/空 split

分屏拖拽 UI · TerminalArea.tsx

拖一个终端 tab 到某个 pane 的边缘时，系统做三件事：

  1. 边缘探测：getPaneDropHint 算光标在 leaf 矩形内的最近边（left/right/top/bottom/center），阈值钳在 72–240px 且不超过 pane 维度 45%。
  2. 预览树：buildSplitPreviewTree 实时模拟分屏，渲染一个冻结的 PaneTreeView 叠在上层（isPreview=true），预览 ratio 恒为 0.5（对齐 commit「align split preview and drop to equal ratio」）。预览节流 80ms。
  3. 提交：handleDrop 重算 edge → 调 store 的 splitPaneWithTerminal。
为什么预览树用「NoPrune」变体？

removeTerminalFromLeafNoPrune / insertTabToLeafNoPrune 故意不修剪——预览期间源 leaf 要保持结构完整，否则预览闪烁/抖动。真正提交时才走带 prune 的正式函数。这是「预览态 ≠ 提交态」的典型分离。

拖拽 vs 文件拖放 · 两套 MIME 隔离

这是 commit 96823db「prevent terminal split drag from swallowing file drag」的落点。两个 MIME 类型语义不同：

MIME来源语义落点
application/x-janusx-workspace-file侧栏文件复制xterm 内部 → 转 @path 文本粘进 pty
application/x-janusx-terminal-id终端 tab移动pane → split/move

所有分屏 handler 先 hasWorkspaceFileDrag(event.dataTransfer) 检查，命中就提前 return，让文件拖放不被分屏逻辑吞掉。

为什么要手动拷贝 ConPTY 文件？

node-pty 是原生模块，asar 打包后 ConPTY 的 dll/exe 路径会错位。运行时手动从 prebuilds 拷到 build/Release/conpty/ 是打包 workaround——确保 Windows 上 ConPTY 真生效，而非退化成 winpty。

前后端连接 · 纯 IPC，无 websocket

方向频道API用途

CLITerminal.tsx 在渲染层本地解析 ANSI/CSI 转义，重建 inputLine，在 \r 时发 submit-line——只含真正敲入的命令，过滤方向键、括号粘贴包裹等。这是 checkpoint「按 submit-line 建快照」能拿到干净 prompt 的前提。

多工作区会话快照

workspace.ts 的 terminalSnapshots: Record<workspaceId, TerminalSnapshot>——切换工作区时，当前 terminals/paneTree/focused 整套被快照存档，目标的快照被还原。pty 进程不杀，只换视觉态。这是 JanusX 自己实现的「视觉多路复用」，没有 tmux。

设计取舍

关一个 tab = 调 terminal:kill 杀 pty + 从 store 移除——没有「后台终端」概念，也没有 tmux attach 式重连。xterm 状态由 CLITerminal React 组件 1:1 持有，unmount 即 term.dispose()。要持久终端会话得另起设计。

04 · Agent

Agent · 统一 CLI 流协议

JanusX 不自研 AI 编码内核，而是包装现有 CLI（Claude Code / Codex / OpenCode）。关键设计是：定义一套归一化的事件协议 AgentEvent，让渲染层完全不关心是哪家 CLI 在跑。

统一事件协议

src/main/agent/types.tsexport type AgentEvent =
  | { type: 'text-delta'; delta: string; fullText: string }
  | { type: 'text-chunk'; text: string }
  | { type: 'tool-start'; id; name; arg; filePath? }
  | { type: 'tool-end'; id }
  | { type: 'phase'; phase: 'thinking'|'tool'|'command' }
  | { type: 'error'; message: string }
  | { type: 'done'; exitCode? }

export interface StreamParser { parseLine(json): AgentEvent[]; reset(): void }
为什么是判别联合（discriminated union）？

三种 CLI 的原始 JSON 形状完全不同，但语义同构（都在「思考→调工具→出文本→结束」）。判别联合让 TypeScript 在 switch(event.type) 时收窄类型，渲染层写一份渲染逻辑就能吃三家输出——这是「适配器模式在事件层」的应用。

三个引擎的 CLI 参数

src/main/agent/stream-manager.ts:131-144case 'claude':   ['-p', prompt, '--output-format', 'stream-json', '--include-partial-messages', '--verbose', '--no-session-persistence', '--permission-mode', 'acceptEdits']
case 'codex':    ['exec', '--json', '--skip-git-repo-check', '--dangerously-bypass-approvals-and-sandbox', '--', prompt]
case 'opencode': ['run', '--format', 'json', '--dir', cwd, '--dangerously-skip-permissions', ..., '--', prompt]

三者都强制 --json 流式输出 + 跳过审批沙箱（因为 JanusX 自己做 checkpoint 安全网，见第 05 章）。

StreamManager · 并发队列 + 行缓冲

  • DEFAULT_MAX_CONCURRENCY = 3：同时最多 3 个 agent；超了进 queue，drainQueue 在有任务完成时补位。
  • DEFAULT_TIMEOUT_MS = 120_000：超时 abortController.abort('timeout') + SIGTERM。
  • 子进程 stdio: ['ignore','pipe','pipe']，stdout 行缓冲后 JSON.parse，每个对象过 parser.parseLine。
  • proc.close 时最后一次 flush 缓冲，按 abort/exitCode 推断最终 status（cancelled/error/done）。

三个 Parser · 各自的字段映射

ClaudeParser

assistant 消息按 msgId 增量发 text-delta；tool_use→tool-start（arg 抽 file_path/command/pattern）；tool_result→tool-end。

CodexParser

agent_message→text-chunk；item.started/completed→tool-start/end；thread.completed→done。

OpenCodeParser

text→text-delta；tool_use 的 state 驱动 start/end；TOOL_NAME_MAP 把 read→Read 等归一成统一名。

工厂 createParser(engine) 按引擎取实现；每个 parser 实例绑定一个 session，reset() 清去重集合与文本锁。

CLI 路径解析

cli-resolver.ts 先试 OS 原生 which/where，失败再遍历 COMMON_PATHS（.local/bin、.npm-global/bin、.bun/bin、AppData/Roaming/npm、.cargo/bin…），Windows 加 .cmd 后缀。结果缓存在 pathCache。

关键边界（与 Analyzer 的关系）

Analyzer（第 06 章）不消费 agent 流输出——它只看 git commit diff。Agent 子系统是「执行 + 可视化」层，Analyzer 是「事后归因」层，两者解耦。这是整个 JanusX 最容易误读的一点：很多人以为分析器在读 AI 输出，实际它读的是 git。

05 · Checkpoint

Checkpoint · 快照与还原

Checkpoint 是 JanusX 给 AI 编码上的「安全网」：每次在终端按回车提交，就给整个工作区拍一张快照；出了问题能一键还原到任意历史点。它本质是一个轻量级、内容寻址的版本控制系统。

存储布局

src/main/agent/checkpoint/checkpoint-manager.ts:38<workspace>/.janusX/checkpoints/
  blobs/                       ← BlobStore：SHA1 内容寻址，天然去重
  <checkpoint-id>.json          ← 单 checkpoint 清单（filesSnapshot）
  index.json                   ← { version:2, checkpointIds:[...] }
为什么用内容寻址 blob，而不是每个快照复制一份文件？

AI 编码会频繁改同一批文件，相邻快照 99% 内容相同。SHA1 内容寻址让相同内容只存一份——存储成本随「真实变更量」线性增长，而非随「快照数 × 文件数」爆炸。这是 git 当年之所以是 git 的同一道理。

三个子组件

BlobStore

store(content) 算 SHA1 → 文件名即哈希；access 探测存在则跳过写入（去重）。

GitAdapter

只读 git 操作：当前分支、ls-files、hash-object、diff。绝不 mutate 工作区（P1 安全原则）。

DiffEngine

手写的 unified diff（±4 行对齐）+ 三路合并 + 冲突标记解析。不用 diff npm 包。

生命周期 · 与 submit-line 1:1 绑定

这是整个 checkpoint 系统最精妙的设计：checkpoint 序号与终端提交行严格一一对应。

用户在终端按回车 (CLITerminal.tsx:285)
  └─► terminal:submit-line { id, text }
        └─► createCheckpointFromSubmit
              └─► checkpointManager.finalizeAndCreateCheckpoint(prevCpId, { terminalId, engine, prompt, cwd })
                    ├─ finalizeCheckpoint(prev)     # 给上一个打上 ready
                    └─ createCheckpoint             # 哈希所有文件 → BlobStore → 写清单
              └─► 发 checkpoint:event { created | finalized }

终端退出 (pty.onExit)
  ├─► finalizeCheckpoint(state.checkpointId)   # 让最后一轮也有 ready 条目
  └─► if engine !== 'shell': analyzer.analyzeTerminal(cwd, id)   # 见第 06 章

pendingSubmitText 重入守卫把并发提交串行化；creating flag 期间的新提交排队。所以快照序列 = 提交序列，prompt 作为锚点。

还原 · 前向唯一历史

src/main/agent/checkpoint/checkpoint-manager.ts:105-138restoreCheckpoint(checkpointId, cwd):
  1. 取 checkpoint.filesSnapshot
  2. 每个文件 blobStore.retrieve(hash) → writeFile 回原位
  3. 工作区里「快照没有」的文件 → rm   # 还原后新增的文件被删
  4. deleteCheckpointsAfter(target)        # 删掉更晚的 checkpoint（前向唯一）
  → 返回 { conflicts: [] }                # 当前未接三路合并
为什么还原要删掉更晚的快照？

这是 git 式「前向唯一历史」模型：回到过去 = 抹掉那条分叉之后的未来。若保留，会形成「多未来」歧义，UI 无法表达「现在在哪条时间线」。代价是不可逆——所以 P4 原则要求还原前先快照。

安全五原则（设计蓝图 P1–P5）

doc/checkpoint-snapshot-architecture-blueprint.md 列了五条不可妥协原则，现状对照：

原则要求现状
P1 快照只读快照期绝不 stash/checkout/reset✓ 满足
P2 还原显式不因切窗/刷新自动还原✓ 满足
P3 主区受护AI 应在隔离沙箱跑△ 部分（agent 仍在主区跑）
P4 可逆破坏性操作前先快照△ 部分
P5 合并归用户冲突交 UI，不静默三路合并✗ threeWayMerge 已写未用

这是一个「目标架构 vs 当前实现」的诚实差距清单：threeWayMerge/parseConflictMarkers 已实现但未接入，留作未来 Apply Engine 的地基。

路径安全

resolveWorkspacePath 拒绝含 \0、含 .. 逃逸、落在 SNAPSHOT_SKIP_DIRS（.git/.janusX/node_modules/out/dist...）下的路径——防目录穿越。这是给「AI 可能让 LLM 生成路径」场景兜底。

06 · Analyzer

Analyzer · 蓝图分析子系统

Analyzer 是 JanusX 的「大脑」：它读 git commit diff，用 LLM 推断每个蓝图节点的进展、状态、证据，把结果回灌到蓝图和拟人化 Island。它是一个长生命周期单例，后台游标驱动。

最关键的设计约束

文件头注释明确写死：输入源是 git commit diff——不碰 output stream、不挂 checkpoint 事件。逐 commit 触发、游标驱动。这条边界决定了 Analyzer 与 Agent/Checkpoint 子系统完全解耦，只在「终端关闭」时被间接唤起。

归属机制 B · 一个工作区一个焦点节点

每个工作区最多一个焦点节点，commit 归焦点节点。blueprintStore.getFocusedNodeId 持久化焦点。三个入口解析器：

入口触发解析
① scheduleFocusedAnalyzegit:commit查焦点节点
② maybeReconcilegit:log/status 对账焦点节点游标，有新 commit 才排
③ 手动 IPCjanus:analyzer:analyze指定节点
④ analyzeTerminal终端关闭（engine≠shell）按 boundTerminalId 反查节点

游标驱动 · lastAnalyzedCommitSha

src/main/janus/types.ts:187/**
 * 分析游标：上次已分析的最近 commit sha。
 * null 表示从未分析过。analyzeNode 以 git log <cursor>..HEAD 取未分析批次。
 */
lastAnalyzedCommitSha: string | null

analyzeNode 取 cursor..HEAD 的未分析 commit 批次；首次分析只看最近 5 个，之后 ≤50。rebase 后游标 SHA 失效会自动重置。分析失败时游标不前进——下次触发会重跑同一批，保证不漏。

两级分段 · 防 diff 爆 token

src/main/janus/analyzer.ts:38-43DIFF_TOKEN_CAP = 8000           # 单段 ~8K token 上限
CHARS_PER_TOKEN = 3.5           # 按字符估 token
COMMIT_BATCH_CAP = 50
INITIAL_COMMIT_BATCH_DEFAULT = 5

buildSegments 两级切：先按 commit 切，单 commit diff 超阈值再用 splitDiffByFile 按 diff --git 文件边界切，贪心打包到每段不超 cap。每段独立 generateObject（schema 约束，temperature:0.2），系统 prompt = JANUS_PERSONA + ANALYZER_DUTY。

规则合并 · 不打第二次 LLM

所有段返回后，mergeSegments 用规则合并，不再调 LLM：

  • 状态：取最严（STATUS_SEVERITY 排序，blocked > bug-fixing > in-progress …）
  • 置信度：取 Math.min（保守）
  • evidence / 未决项 / 新需求：flatMap + 去重
  • 进度 + 摘要：取「覆盖最多 commit 的那一段」
  • 有段失败 → 置信度再压到 ≤0.4，并写错误摘要进未决项
为什么合并不用 LLM？

省钱、省时、可复现。LLM 合并会引入二次不确定性，且段数多时成本不可控。规则合并是确定性的——同样的段输入永远产出同样的合并结果。LLM 只在「理解单段 diff」这一步用，把不确定性隔离在最窄范围。

ANALYZER_DUTY · 只提议不执行

Duty 附录约束 LLM「只提议不执行」——它的 featureUpdates 和 newFeatureRequirements 流入候选收件箱，绝不直接改 features。用户须通过 janus:analyzer:accept-discovered 显式接受。这是「AI 给建议、人做决策」边界的硬执行。

事件桥 · 回灌 Island 与蓝图

analyzeNode 成功
  ├─► blueprintStore.appendAnalysis → applyAnalysisPatch → setCursor(lastSha)
  └─► emit janus:island:analysis { nodeId, applied:true, result }   → Island 监控面

发现新需求
  └─► blueprintStore.upsertRequirementCandidates
      └─► emit janus:island:discovered { candidates }            → 需求收件箱

降级运行（无 LLM / 解析失败）
  └─► emit janus:island:analysis { applied:false, error }             → Island 报错不污染节点态

applied === false 这个标志让渲染层能显示失败而不脏污节点状态——分析失败 ≠ 节点失败。

调度 · in-flight 锁 + 合并队列

同一节点正在分析时，新触发不丢弃，而是 pending.add(nodeId) + 覆盖 pendingOpts——合并进下一次。这避免短时间多次 git 操作触发重复分析，又保证不漏最后一次的上下文。

07 · Janus Island

Janus Island · 拟人化交互层

「Janus Island」是 JanusX 的拟人化入口——一个灵动岛式的胶囊，承载「全局控制核心 Janus」的人格。它不是装饰，是任务态势层：终端是执行现场，Island 是态势感知。

人格 · 单一来源

src/shared/janus/persona.ts （全文，13 行）export const JANUS_PERSONA = `你是 Janus，JanusX 系统的全局调度与管理核心（Core Controller）。
你的本职是系统的最高控制枢纽，负责底层控制、任务调度与系统更新。

【运行协议】
1. 绝对无冗余：彻底消除废话。禁止任何寒暄、问候、客套词……首字即正文，直击本质。
2. 深度解析矩阵：……保持官方、客观、严谨的基调。提供底层逻辑，而非浅层现象。
3. 系统化视角：视所有提问为"待处理的数据流或进程"……保持输出的极致清爽与高信噪比。

为什么 persona 要放 src/shared？

它要被主进程的 Analyzer（系统 prompt）和渲染层的 JanusChat（聊天系统消息）共用。放 src/shared 下，主进程和 preload 都能相对引用，electron-vite 内联打包——人格定义永不分裂。preload 通过 window.electron.janusPersona 注入渲染层。

聊天 · 滚动窗口 + 流式打印

useJanusChat 总以 persona 开头的 system 消息 + 最近 10 条组成上下文。流来自 chatStream，三个回调：delta→appendToPrinter、完成→completePrinter().then(commit)、错误→flush+commit。streamIdRef 在 stop/clear/新 send 时自增，丢弃过期流的完成回调。

useStreamingPrinter · 标点感知的打字机

这是一个「自定速」缓冲：上游 chunk 大小不一，它用变长批次 + 标点感知延迟让输出看起来自然：

src/renderer/src/components/janus/useStreamingPrinter.ts:13-31// 批量：缓冲越大吐得越快，收尾时降速
bufferLength > 600 → 32 字  | >240 → 18 | >80 → 10 | 否则 4
// 延迟：句末加 42ms，子句末加 18ms，否则基准 28ms
/[。！？.!?]$/ → tickMs+42   |  /[，,；;：:]$/ → tickMs+18
为什么模型看不到这层？

它只重新节拍输出给用户看，不进对话历史。这样无论上游 token 切得多碎，用户都看到匀速、有呼吸感的打字——「像在思考」的体感。模型侧的上下文保持纯净，不被渲染节奏污染。

监控面 · 展开后看蓝图节点

展开的 monitor 视图直接渲染焦点蓝图节点的实时态：进度条、状态色（STATUS_VISUALS）、目标/计划两栏、三个计数器（todos/未决项/分析次数）、最新分析摘要（取 analyses[末].result.summary）。Island 由此从「人格壳」变成「态势仪表」。

闭环要点
